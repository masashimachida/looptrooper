# draft-loop-issue

人間（プランナー役）と対話して、**ループの executor (loop-task) が高品質な PR を出せる issue** を1本作り、`gh` で起票するまでをやるスキル。

原則: **issue の質がループ出力の質を決める**。曖昧な issue → 凡庸な PR。対象 repo に接地した・検証可能な issue → ループが自己検証まで到達する（実証済み）。
あなたは「**何をやるか**」を決めない —— それは人間が決める。あなたの仕事は、人間が決めた意図を **正確・最小・検証可能** な形に落として起票すること。

> このスキルは **人間が対話的に使う**もので、unattended なループ本体（`loop-task`）とは別物。`setup-target` では配布されない。
> **前提: 対象 repo の checkout の中で実行する**（ローカルのファイルを読んで接地し、その repo に起票するため）。

## 大前提（loop-task との契約）

立てた issue は `loop-task` が処理する: worktree 隔離 → Fixer 実装 → **別**サブエージェント(Verifier)が `$BUILD_CMD`/`$TEST_CMD`/`$LINT_CMD` ＋ `/code-review` で検証 → `loop/<id>` に push → PR 提案（**マージは必ず人間**）。だから issue は:

- **1 issue = 1 目的 = 小さい1 PR**。大きい要望は分割し、順序があれば GitHub の "blocked by"（issue dependency）で繋ぐ（`loop-task` はこれを尊重し、ブロック中は着手しない）。
- **受け入れ条件は機械的に検証可能**に。特に **「実装後に実際に `<コマンド>` を走らせて green を確認する」** を最低1つ入れる —— これで Verifier が回さない種類の検証でも Fixer が自分で実行する（強力なテク）。
- **既知の制約・落とし穴を先に書く**（Fixer に再発見させない）。

## 手順

### 1. 意図を掴む
人間のざっくりした要望を聞く。**勝手にスコープを足さない。**

### 2. 対象 repo に接地する（最重要）
issue を書く前に**実コードを読む**。最低限:
- `package.json`（scripts / deps）、`docker-compose*` / `Dockerfile`、関連設定（lint/test/tsconfig 等）、変更対象のファイル/ディレクトリ。
- ビルド/テスト/起動の癖、サービス名・ポート・依存（DB 等）。

接地して初めて「具体的な変更手順」「現実的な受け入れ条件」「落とし穴」が書ける。**推測で書かない。**

### 3. 詰める（曖昧点だけ質問）
本当に曖昧な所だけ人間に確認する: スコープ境界 / 受け入れ条件 / 触ってはいけない範囲 / 非機能要件。
- 1 PR に収まらないなら**分割を提案**（複数 issue ＋ 依存関係）。
- 既存挙動を壊す変更（リネーム・I/F 変更等）は**影響範囲を洗って明記**。

### 4. 起案（この型で書く）
```
タイトル: <動詞で始まる簡潔な要約>

## 目的
<なぜ・背景。1〜3文>

## 変更（最小・1PR）
- <ファイル/箇所レベルの具体手順>
- <既存の何を「変えない」かも明記>

## 受け入れ条件
- <機械的に検証可能な条件>
- 実装後に `<検証コマンド>` を実行して green を確認する
- 既存のビルド/テストを壊さない

## 備考
- <接地して見つけた制約・落とし穴・前提（例: このサービスは alpine なので X 不可、ホスト名 Y は HSTS で HTTPS 強制、等）>
```

### 5. 人間の承認を取る
ドラフト全文を見せ、**明示的な OK をもらってから**起票する。**勝手に立てない。**

### 6. 起票
```bash
gh issue create -R "<owner>/<repo>" --label loop \
  --title "<title>" --body "<body>"
```
- repo は **ローカル checkout の origin から導出**する（`gh repo view --json nameWithOwner -q .nameWithOwner` など）。
- 既定は **`--label loop`**（実装〜PR まで自走）。人間がもう一段ゲートを置きたい時は **`--label loop:proposed`**（人間が後で `loop` を付けて承認したら着手）。
- 分割した依存付き issue は、起票後に "blocked by" を設定（GitHub UI、または `gh api` の `repos/{owner}/{repo}/issues/{n}/dependencies/blocked_by`）。
- 出力された **issue URL を人間に返す**。

## やってはいけない
- 人間が頼んでいないスコープを足す / 「何をやるか」を勝手に決める。
- **承認前に起票する。**
- 接地せず推測で書く。
- repo のコードを**実装・変更**する（ここは**起票だけ**。実装はループの仕事）。
