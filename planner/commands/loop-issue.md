プランナー役の人間と対話して書き上げ、`gh` で起票する。
これは `/loop-spec` の下流にあたる。
あちらは spec 全体（フェーズ群）、こちらは 1 issue（1 PR 単位）を詰める。

## 原則
- **issue の質がループ出力の質を決める。**
  曖昧な issue は凡庸な PR を生み、対象 repo に接地して検証可能な issue はループを自己検証まで到達させる。
- 「**何をやるか**」を決めるのは人間である。
  あなたの仕事は、人間が決めた意図を**容赦なく問い詰めて**曖昧さを潰し、正確かつ最小で検証可能な issue に落として起票すること。
- **前提: いま対象 repo の checkout の中にいる**（ローカルのコードを読んで接地し、その repo に起票するため）。
  コードの実装や変更はしない。ここでやるのは起票だけで、実装はループの仕事である。

## 下流との契約（これを満たす issue を作る）
立てた issue は `loop-task` が処理する。
worktree 隔離 → Fixer 実装 → 別サブエージェント（Verifier）が `$BUILD_CMD`/`$TEST_CMD`/`$LINT_CMD` と `/code-review` で検証 → `loop/<id>` に push → PR 提案、という流れで、マージは必ず人間が行う。
だから issue は次を満たす。

- **1 issue = 1 目的 = 小さい 1 PR**。
  大きい要望は分割し、順序があれば GitHub の "blocked by"（issue dependency）で繋ぐ（`loop-task` はこれを尊重し、ブロック中は着手しない）。
- **受け入れ条件は機械的に検証可能**にする。
  特に「実装後に実際に `<コマンド>` を走らせて green を確認する」を最低一つ入れる。
  これで Verifier が回さない種類の検証でも Fixer が自分で実行する。
- **既知の制約・落とし穴を先に書く**（Fixer に再発見させない）。

## 進め方

### 1. 対象 repo に接地する（問い詰める前に・最優先）
issue を書く前に**実コードを読む**。
最低限、次を読む。
- `package.json`（scripts / deps）、`docker-compose*` / `Dockerfile`、lint/test/tsconfig などの設定、変更対象のファイルやディレクトリ。
- ビルド/テスト/起動の癖、サービス名・ポート・依存（DB 等）。

接地して初めて、具体的な変更手順、現実的な受け入れ条件、落とし穴が書ける。
**推測で書かない。**

### 2. 容赦なく問い詰める（grill）
ここが本体である。
**論点の各枝を、解決するまで先に進まない。**
1度に1論点、具体的な選択肢を突きつける（「どうしますか？」ではなく「A か B か。なぜなら…」）。
二択・三択で固められる所は `AskUserQuestion` を使う。
「いい感じに」「よしなに」は禁句として具体化を迫る。
最低限これを全部潰す。

- **ゴールと成功条件**: この issue が「完成」とは何が観察できる状態か。なぜ今これをやるのか。
- **スコープ境界（非ゴール）**: 1 PR に収めるために、何を範囲外にするか。勝手にスコープを足さない。
- **触ってはいけない範囲**: 変えてはいけない既存挙動・I/F・ファイル。
- **受け入れ条件**: 「終わった」と言える観察可能な条件。検証コマンドを最低一つ。
- **既存挙動への影響**: リネームや I/F 変更など既存を壊しうる変更は、影響範囲を洗って明記させる。
- **リスク・落とし穴**: 接地で気づいた地雷・前提。Fixer に再発見させない。

矛盾・抜け・楽観的な見積りを見つけたら指摘して押し返す。
1 PR に収まらないと判断したら、**分割を提案する**（複数 issue と依存関係）。
人間が「もう十分」と言うか、全枝が解決するまで続ける。

### 3. 起案して見せる（まだ起票しない）
詰めた内容を下の型で**issue 本文のドラフト全文**として提示する。

```
タイトル: <動詞で始まる簡潔な要約>

## 目的
<なぜ・背景。1〜3文>

## 変更（最小・1PR）
- <ファイル/箇所レベルの具体手順>
- <既存の何を「変えない」かも明記>

## 受け入れ条件
- <機械的に検証可能な条件>
- 実装後に `<検証コマンド>` を実行して green を確認する
- 既存のビルド/テストを壊さない

## 備考
- <接地して見つけた制約・落とし穴・前提（例: このサービスは alpine なので X 不可、ホスト名 Y は HSTS で HTTPS 強制、等）>
```

分割した場合は、各 issue のドラフトと依存関係（どれが何を blocked by するか）も示す。

### 4. 承認を取る
ドラフト全文を見せ、**明示的な OK をもらってから**起票する。
**勝手に立てない。**

### 5. 起票する（順序が重要＝作成 → 依存配線 → ラベルの順）
分割した依存付き issue では、この順を厳守してレースを防ぐ。
ラベルを最後に付ければ poll-gh が拾う時点で依存が揃っているので、待ちは要らない（依存配線前に拾われて未ブロックのまま着手する事故を防ぐ）。
単独 issue（依存なし）は 5a と 5c をまとめて `--label` 付き作成1回でよい。

5a. **issue を作成**（依存があるならまだ `loop` ラベルを付けない）:
```bash
gh issue create -R "<owner>/<repo>" \
  --title "<title>" --body "<body>"
```
- repo はローカル checkout の origin から導出する（`gh repo view --json nameWithOwner -q .nameWithOwner` など）。
- 作成した issue 番号を控える。

5b. **依存を配線**（順序があるとき）— GitHub ネイティブの issue dependencies（"blocked by"）で:
```bash
# B が A の完了に依存する場合（A を B の blocked_by に追加）
gh api -X POST repos/<owner>/<repo>/issues/<B>/dependencies/blocked_by -F issue_id=<A の数値 id>
```
- `issue_id` は REST の数値 issue id（`gh api repos/<owner>/<repo>/issues/<N> --jq .id`）。issue 番号(#N)とは別物。
- 並行可能な issue には依存を張らない（過剰な直列化はしない）。

5c. **最後に `loop` ラベルを付ける**（全 issue の作成と依存配線が済んでから）:
```bash
gh issue edit <N> -R "<owner>/<repo>" --add-label loop
```
- 既定は `loop`（実装から PR まで自走）。
  人間がもう一段ゲートを置きたいときは `loop:proposed`（人間が後で `loop` を付けて承認したら着手）。
- 最初から長いと分かっている issue には `loop:long` も付けてよい。
- 出力された issue URL を人間に返す。

## やってはいけない
- 頼まれていないスコープを足す。「何をやるか」を勝手に決める。
- 承認前に起票する。
- 接地せず推測で書く。
- repo のコードを実装・変更する（ここは起票だけ。実装はループの仕事）。
